topic: Add POSIX-only directory fsync after atomic-write rename, skipped on Windows
updated: 2026-07-17T00:29
---

- (note) Source: deferred-work.md entry under spec-atomic-file-writes.md - src/jobs/cli.py's _atomic_write_text (line 239) and src/jobs/docx_tailor.py's _atomic_write_bytes (line 44) both fsync the temp file's data before os.replace(), but never fsync the containing directory after the rename. A crash immediately after os.replace but before the OS flushes directory metadata can still lose the rename on POSIX.
- (note) Original review declined this: 'doing it portably (POSIX directory-fd fsync has no direct Windows equivalent) is real added complexity beyond matching the existing, already-audited memlog.py precedent.' Verified empirically on this Windows dev machine: os.open(dir, os.O_RDONLY) raises PermissionError, confirming no direct Windows equivalent for this exact operation.
- (decision) The fix doesn't need to be portable, just POSIX-only behind an os.name == 'posix' guard - a well-known standard pattern, a no-op skip on Windows (matching current behavior exactly there). This project's actual deploy targets include Streamlit Community Cloud (Linux containers), so this isn't purely theoretical for the code even though the local dev machine is Windows - that's exactly why the gap was never POSIX-tested before.
- (capability) CAP-1 intent: on POSIX, _atomic_write_text's temp-then-rename durability guarantee extends to the directory entry itself (not just the file's data), so a crash immediately after the rename can't lose it. success: on a POSIX system, mocking/monitoring the write path confirms the parent directory is opened and fsync'd after a successful os.replace(); on Windows this step is skipped entirely (verified: no behavior change, no error) - both verified via unit tests using os.name mocking rather than requiring a real POSIX machine to run the suite.
- (capability) CAP-2 intent: the identical fix applies to _atomic_write_bytes (docx_tailor.py), which mirrors _atomic_write_text's design for binary content. success: same as CAP-1, verified against _atomic_write_bytes's own code path.
- (constraint) Guard with os.name == 'posix', not a try/except swallowing the Windows PermissionError - an explicit platform check is clearer than relying on a caught exception to distinguish 'not supported here' from 'a real failure on a POSIX system'.
- (constraint) A directory-fsync failure on POSIX propagates as OSError, matching how a failure earlier in the same function already propagates - the whole point of this helper is a durability guarantee, so a failure in the new step must not be silently swallowed.
- (constraint) Both _atomic_write_text (jobs/cli.py) and _atomic_write_bytes (docx_tailor.py) get the identical fix, matching their existing mirrored-design relationship (docx_tailor.py's docstring already says it mirrors cli.py's design) - not diverging the two helpers.
- (non-goal) No attempt at a Windows equivalent (e.g. via ctypes/win32 flush APIs) - out of scope; Windows behavior is unchanged (still only file-data fsync, no directory-level guarantee), matching today exactly.
- (non-goal) No change to the temp-file naming, cleanup-on-failure, or any other part of the existing atomic-write pattern - purely additive, one new step after a successful rename.
