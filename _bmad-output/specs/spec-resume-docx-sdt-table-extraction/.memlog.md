- (decision) Source: deferred-work.md entry under spec-resume-docx-sdt-extraction/SPEC.md - a w:sdt wrapping a single table row or an in-cell paragraph is still dropped, one level deeper than the body-level w:sdt fix already shipped.
- (decision) Investigated python-docx internals empirically (real generated XML, not guessed): _Rows.__iter__ and _Row.cells only see direct w:tr/w:tc children. A horizontal merge (.merge()) physically collapses to one w:tc with w:gridSpan - the other w:tc is deleted from the XML - so a walker visiting each physical w:tc once needs no merge dedup (the current _row_text's seen_tc dedup exists only to undo iter_tc_cells' artificial gridSpan grid-expansion).
- (decision) Investigated vertical merges: both w:tc elements stay physically present; the continuation row's w:tc has vMerge and an empty w:p. Confirmed via live repro that current output repeats the real content in both rows (e.g. Skills|A newline Skills|B) - this exact behavior must be preserved.
- (decision) Critical correctness trap found: CT_Tc.grid_offset counts only DIRECT preceding w:tc siblings via xpath - undercounts when a w:tc's real parent is w:sdtContent (sdt-wrapped), giving a wrong _tc_above resolution, but only for a cell that is BOTH sdt-wrapped AND itself vMerge=continue. _tr_above is ancestor-based and IS sdt-safe - only grid_offset's sibling-counting breaks.
- (decision) Design chosen for the trap: vMerge=continue resolution only attempted when the cell is a direct child of its w:tr (tc.getparent().tag == qn(w:tr) - the well-tested pre-existing-correct path). An sdt-wrapped cell is shown with its own literal content instead of resolving vMerge continuation - accepted as a narrow, documented residual (real Word templates wrap actual content in controls, not merge-continuation placeholder cells).
- (capability) CAP-1: A w:sdt wrapping a whole table row (w:tr) is read - its cells' text appears in the table's output in correct document order, same as an unwrapped row. Success: a table with one sdt-wrapped row interleaved with unwrapped rows extracts all rows, in order.
- (capability) CAP-2: A w:sdt wrapping a single cell (w:tc) within an otherwise-normal row is read, in its correct position among the row's other cells. Success: a row with one sdt-wrapped cell and unwrapped siblings extracts all cells' text in the correct left-to-right order.
- (capability) CAP-3: A w:sdt wrapping a single paragraph inside a table cell (alongside other unwrapped paragraphs in the same cell) is read as part of that cell's text. Success: a multi-paragraph cell with one paragraph sdt-wrapped includes that paragraph's text in the cell's newline-then-space-flattened output.
- (capability) CAP-4: All existing table-extraction behavior is preserved exactly - horizontal-merge collapsing (one appearance, not duplicated), vertical-merge content repetition across rows for non-sdt-wrapped cells, the all-blank-row skip, multi-paragraph-cell newline-to-space flattening, and the depth-capped iterative (non-recursive) walk. Success: the full existing test_resume_extract.py suite (pre-this-change) still passes unmodified.
- (constraint) A generic, shared sdt-unwrapping traversal helper backs all levels (body, row, cell, paragraph-within-cell) - not four near-duplicate stack-walk implementations. The existing body-level _iter_body_content_including_sdt is refactored to build on this same generic core.
- (constraint) Vertical-merge (vMerge=continue) resolution is only attempted for a cell that is a direct child of its w:tr - never for an sdt-found cell, per the grid_offset correctness trap. This is a narrow, explicit design boundary, not an oversight.
- (non-goal) Not resolving vMerge continuation for a cell that is itself sdt-wrapped - documented residual (see the constraint above), not attempted given how rare the combination is in real documents.
- (non-goal) Not nested tables inside a cell, headers, footers, or text boxes - already out of scope per the body-level fix and the original table-extraction spec, unchanged by this spec.
- (event) Self-validate: Pass 1 (coherence) OK - 4 capabilities each with intent+success (CAP-4 explicitly guards no-regression), constraints bend design (shared generic helper, vMerge-resolution boundary), 2 non-goals, concrete success signal covering row+cell+paragraph wrapping together. Pass 2 (preservation): deferred-work.md claim and all investigation findings (merge-collapse behavior, vMerge repetition, the grid_offset trap and its resolution) land in SPEC.md's Why/Capabilities/Constraints. No wrapper-only content dropped. SPEC.md finalized.
- (event) Implementation-time discovery via live testing (not review): the spec's original constraint ('vMerge resolution only breaks when the RESOLVING cell itself is sdt-wrapped') was incomplete. Empirically confirmed wrapping the CONTENT-HOLDING (top) cell of a vertical merge also breaks resolution - and worse, it doesn't fail loudly: CT_Tc.grid_offset's search inside the row above (tc_at_grid_offset, direct-children-only) silently resolves to the WRONG cell (a sibling's content) instead of raising, once the real target cell is no longer a direct child.
- (decision) Redesigned the guard to be conservative rather than narrowly-scoped: vMerge=continue resolution is now only attempted when BOTH the current row and the row above have every cell as a direct <w:tr> child (no w:sdt wrapping anywhere in either row) - via a new _row_cells_are_all_direct_children(tr_element) check applied to both rows. This guarantees the silent-wrong-data failure mode can never occur, at the cost of a slightly wider residual (an sdt-wrapped cell ANYWHERE in a vMerge chain, not just the specific continuation cell, disables resolution for that chain) - mutation-tested by removing the two _row_cells_are_all_direct_children checks and confirming the wrong-cell-resolution bug reproduces exactly as first observed.
- (event) Self-validate (re-run after guard redesign): Pass 1 (coherence) OK - constraints updated to state the stronger, both-rows-checked guard and why (silent-wrong-data risk, not just unreachability); non-goals updated to describe the resulting residual accurately (any nearby sdt-wrapping, not just the resolving cell). Pass 2 (preservation): the implementation-time discovery and corrected design are both reflected in SPEC.md's Constraints/Non-goals, not left only in the memlog. 298/298 tests passing. SPEC.md re-finalized to match the actually-shipped code.
- (event) Third adversarial review found no correctness bugs in the whole-table redesign (confirmed via its own independent mutation test and live repro verification). Findings were documentation/coverage gaps: patched docstrings for _table_has_any_sdt_wrapping (scope limited to w:sdt specifically, not w:ins/customXml; depth-cap interaction; nested-table safety) and extract_text_from_docx (a suppressed vMerge cell renders indistinguishably from a genuinely blank source cell - accepted, documented). Added a test demonstrating the actual whole-table tradeoff (wrapping one merge's cell disables repetition for a second, unrelated merge elsewhere in the same table) and a zero-row-table edge case test. Both new tests mutation-tested.
