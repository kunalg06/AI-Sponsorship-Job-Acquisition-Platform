- (note) Source: deferred-work.md entry under spec-admin-destructive-safety.md - views/admin.py's CV-upload idempotency guard (last_registered_cv_file_id, checked at line 172, set at line 132/211) keys off uploaded_file.file_id, a Streamlit upload-widget identity stable only for the file currently held by that widget instance. Re-selecting the identical file via the OS file picker produces a new file_id and bypasses the guard - a genuine duplicate profile row and a second Gemini charge.
- (note) Original review called the real fix (hashing raw_text) a design tradeoff since it requires eagerly extracting text on every render just to compute the comparison key, changing error-timing behavior (a corrupt file would fail immediately on upload rather than after clicking the button).
- (decision) That tradeoff doesn't apply if the hash is computed over the raw uploaded bytes, not the extracted text. Verified via dir(UploadedFile): getvalue() returns the full raw byte buffer without consuming the read cursor (unlike .read(), already used by the extraction code at line 197, which already needs a compensating uploaded_file.seek(0) at line 193). hashlib.sha256(uploaded_file.getvalue()).hexdigest() is a pure in-memory byte digest - no parsing, no corruption risk, no possible failure mode - so it introduces no error-timing change versus today's file_id check.
- (capability) CAP-1 intent: the CV-upload idempotency guard recognizes a re-selected file as already-registered by its actual content, not by the upload-widget's transient identity, so re-picking the identical file via the OS file picker no longer bypasses the guard. success: uploading a file, registering it, then re-selecting the byte-identical file via a fresh file_uploader interaction (a new file_id, same bytes) shows the same already-registered state (button disabled, info message) that re-clicking the still-held original upload already shows today.
- (constraint) Hash the raw uploaded bytes via uploaded_file.getvalue(), not the extracted text - must not introduce any new failure mode or change when in the flow a failure can occur (matches today's exact error-timing).
- (constraint) Session-state key renamed from last_registered_cv_file_id to a hash-based name (e.g. last_registered_cv_hash) and used consistently at both the already_registered check and the mark-as-registered step - no site keeps comparing against file_id.
- (constraint) Existing double-click-guard behavior (cv_registration_in_progress gating, the disabled= re-check comment at line 185-189) is unchanged - this only changes what identity the guard compares, not the guard mechanism itself.
- (non-goal) No change to the supersede-profile confirmation dialog or its own dismiss/idempotency handling (spec-cv-supersede-dialog-dismiss-fix's scope) - orthogonal concern, this only changes how a duplicate upload is detected before that dialog is ever reached.
- (non-goal) No persistence of the hash beyond the current session_state (e.g. no DB column recording every registered file's hash) - matches today's scope (last_registered_cv_file_id is also session-only, reset on a fresh session).
