---
topic: _write_tailoring_files rolls back the resume file if the cover-letter write fails, so the pair is never left mismatched-but-looking-complete
updated: 2026-07-17T16:31
---

- (decision) Source: deferred-work.md entry under spec-atomic-file-writes.md - _write_tailoring_files (jobs/cli.py:297) writes resume then cover-letter as two independent atomic writes; if the second fails after the first succeeds, a mismatched pair (new resume + stale/old cover letter, or absent on first run) is left with nothing distinguishing it from a genuine success.
- (decision) Investigated: this is _cmd_tailor's write path, deliberately uncached (confirmed via its own docstring) - always calls the LLM fresh and always overwrites both files. So true rollback (restoring the OLD resume's prior content) isn't needed for correctness - re-running tailor after a failure regenerates both files fresh. _atomic_write_text's own contract guarantees a failed write leaves ITS OWN target untouched (full prior content or absence, never truncated), so if the FIRST write (resume) fails, nothing needs to happen - cover_letter_path was never touched. Only the SECOND write (cover letter) failing after the first succeeded creates the actual mismatch.
- (decision) Design chosen: delete the just-written resume file (best-effort) when the cover-letter write fails, restoring the invariant 'after this function returns successfully, both files reflect this call's output; if it raises, neither file exists.' Deliberately not a two-phase-rename/manifest system that would restore old content - that's the bigger design change the original deferred item explicitly ruled out of scope, and isn't needed given the always-uncached, always-regenerate-on-retry nature of this command.
- (capability) CAP-1: When the cover-letter write fails after the resume write already succeeded, _write_tailoring_files deletes the just-written resume file (best-effort) before propagating the original exception, so no half-written pair is left looking complete. Success: simulating a cover-letter write failure after a successful resume write - the resume file does not exist afterward, and the original exception still propagates uncaught (matching this function's existing no-try/except convention for the underlying failure).
- (capability) CAP-2: If the FIRST write (resume) itself fails, no rollback action is needed or taken - both paths are already left exactly as they were before the call (per _atomic_write_text's own contract), since the cover-letter write was never attempted. Success: simulating a resume write failure - the cover-letter file (if one existed from a prior run) is untouched, and no rollback code path runs.
- (constraint) The rollback (deleting resume_path) must itself be best-effort - it must not raise a NEW exception that replaces or masks the original cover-letter write failure the caller needs to see, matching the established pattern this session used for _fsync_directory and the outreach write-failure marker.
- (constraint) Must not change this function's existing no-try/except, let-it-propagate convention for the underlying write failure itself - only a rollback side-effect (and, if feasible without complicating the propagation, a clearer message) wraps around it; the original OSError/ValueError from the cover-letter write must still surface to the caller uncaught, same as today.
- (non-goal) Not a two-phase-rename or manifest system that restores the OLD resume content on cover-letter failure - not needed given _cmd_tailor is always uncached and a retry regenerates both files fresh from a new LLM call; explicitly the 'bigger design change' the original deferred item ruled out of scope.
- (non-goal) Not touching the docx path (_tailor_docx_for_job/build_tailored_docx/write_plain_docx) - a separate write pair with its own cache-check semantics, out of scope for this specific deferred-work entry, which named _write_tailoring_files (the plain-text tailor command's path) specifically.
- (event) Self-validate: Pass 1 (coherence) OK - 2 capabilities each with intent+success, constraints bend design (best-effort rollback, preserve propagation convention), 2 non-goals, concrete success signal. Pass 2 (preservation): deferred-work.md claims land in SPEC.md (mismatched-pair risk -> Why; rollback-on-second-failure -> CAP-1; no-op-on-first-failure -> CAP-2; no manifest/two-phase system, docx path excluded -> Non-goals). No wrapper-only content dropped. SPEC.md finalized.
- (event) Round-1 adversarial review found the CAP-1/CAP-2 design below (delete the resume file on cover-letter failure) was itself buggy: on a re-run (job already tailored before, both files pre-existing), deleting the just-written resume leaves 'no resume + stale old cover letter' - strictly worse than the original bug, since the fresh resume text is gone and nothing restores the cover letter's own prior content without the manifest/two-phase system this spec's Non-goals explicitly excluded. The premise behind CAP-1 ('no old content worth restoring') only held for a first-ever tailor of a job_id, not a re-run.
- (decision) Redesigned: no file deletion at all. On cover-letter write failure, both files are left exactly as they land - resume_path holds this call's fresh text, cover_letter_path is untouched (absent on a first run, or the older content from the last successful write on a re-run, per _atomic_write_text's own contract of never touching its target on failure). Both pieces of freshly-generated text are printed to stdout for manual recovery (mirroring _draft_and_store_outreach's identical situation), and a SystemExit names precisely which file is fresh and which is stale/absent, directing the operator to re-run tailor (always-uncached, regenerates both fresh).
- (capability) CAP-3 (supersedes CAP-1): When the cover-letter write fails after the resume write already succeeded, _write_tailoring_files leaves both files exactly as they land (resume fresh, cover letter untouched), prints both pieces of generated text for manual recovery, and raises SystemExit naming which file is fresh and which is stale/absent - never deletes anything. Success: simulating a cover-letter failure on both a first run (cover letter absent before and after) and a re-run (cover letter's old content unchanged) - in both cases the resume holds this call's fresh text, the cover letter is provably untouched, and the printed output contains both generated texts.
- (event) Self-validate (re-run after design reversal): Pass 1 (coherence) OK - CAP-3 (superseding CAP-1/CAP-2) has intent+success, constraints bend design (no fabricated restoration, resume-branch behavior unchanged, documented exception-type asymmetry), 2 non-goals, concrete success signal now covering both first-run and re-run cases plus a retry. Pass 2 (preservation): the rejection story and the corrected design are both now in SPEC.md's Why/Capabilities, not left only in the rendered-away memlog. SPEC.md re-finalized to match the actually-shipped code.
