---
topic: prevent two concurrent tailoring runs for the same job from racing on the same docx/DB writes
updated: 2026-07-17T10:18
---

- (decision) Source: deferred-work.md entry under spec-force-regenerate-control.md - two browser tabs/sessions both clicking Regenerate for the same job both bypass the cache, both call the LLM, both overwrite the same docx files/DB tailoring columns, last write wins. User explicitly chose this item to tackle next (via AskUserQuestion) over three other design-decision items.
- (decision) Investigated _tailor_docx_for_job (jobs/cli.py:468) - shared by both jobs.cli tailor-docx and ui_actions.generate_tailored_docx_for_job, so a single fix there protects CLI-vs-UI and UI-vs-UI concurrent calls alike, not just the two-tabs case. The cache-check itself (inside _get_or_generate_tailor_text) is also racy (TOCTOU: both callers can see a miss before either writes), so the lock must wrap the whole function body, not just the write step.
- (decision) Streamlit's st.session_state is per-browser-session, not shared across tabs/sessions, so an in-memory/session-state guard cannot prevent the two-tabs race - the guard must live in the jobs DB itself (the one piece of state already shared across all callers of this codebase's single-SQLite-writer model).
- (capability) CAP-1: A second concurrent call to _tailor_docx_for_job for the same job_id, while a first call is still generating, is rejected with a clear SystemExit message instead of proceeding to a real LLM call/file write. Success: two sequential calls simulating overlap (first call's lock claimed and not yet released) - the second raises SystemExit naming the job and stating another session is already regenerating it; no LLM call or file write happens on the rejected call.
- (capability) CAP-2: The lock is always released after the call finishes, whether it succeeds or raises (e.g. an LLM error), so one failed generation never permanently blocks all future retries for that job. Success: a call that raises mid-generation still leaves the job's lock cleared afterward, and an immediately-following call can claim it.
- (capability) CAP-3: A lock left set by a crashed/killed process (never reaching the release step) is reclaimable once it goes stale, rather than blocking that job's tailoring forever. Success: a lock claimed more than the staleness window ago is treated as claimable by a new call, even though nothing explicitly released it.
- (constraint) The lock must live in the jobs DB (a new nullable timestamp column on the jobs table, following this codebase's existing _ensure_columns auto-migration convention), not in-process/session state, since it must be visible across Streamlit sessions and CLI invocations alike.
- (constraint) Claiming must be a single atomic UPDATE ... WHERE (conditional on the column being NULL or stale) so two concurrent claims can't both succeed - a read-then-write check-and-set has the same TOCTOU race this spec exists to close.
- (constraint) The lock wraps the entire _tailor_docx_for_job body (cache-check included), not just the fresh-generation branch, since the cache-check's own hit/miss decision is itself part of the race.
- (non-goal) Not a real distributed lock or job queue - a simple advisory DB flag matches this app's existing single-SQLite-writer, best-effort concurrency mitigations (e.g. register.db's busy_timeout PRAGMA for the sponsor-register race).
- (non-goal) Not queueing/retrying the rejected call automatically - it surfaces as a SystemExit, exactly like every other failure on this path (views/intake.py and views/jobs_list.py already catch SystemExit via error_display_text), and the user decides whether to retry.
- (assumption) Staleness window assumed to be 5 minutes (300s) - long enough to cover a slow LLM call plus docx writes without false-reclaiming a live lock, short enough to recover reasonably fast from a crashed process. Not measured against real generation times; a defensive default, not a tuned value.
- (event) Self-validate: Pass 1 (coherence) OK - 3 capabilities each with intent+success, constraints all bend design (DB-not-session-state, atomic UPDATE, whole-function-not-just-write-step), 2 non-goals, concrete success signal, 1 assumption (unmeasured 300s staleness default). Pass 2 (preservation): deferred-work.md claims land in SPEC.md (two-tabs race -> Why/CAP-1; last-write-wins data loss -> Why; crash-safety -> CAP-3; queueing/distributed-lock explicitly out per this codebase's existing best-effort precedent -> Non-goals). No wrapper-only content dropped. SPEC.md finalized.
