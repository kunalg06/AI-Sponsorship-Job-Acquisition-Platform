topic: mark and distinguish an outreach_messages row whose file write is known to have failed
updated: 2026-07-17T10:41
---

- (decision) Source: deferred-work.md entry under spec-atomic-file-writes.md - a CAP-3 write failure leaves a permanently orphaned outreach_messages row (metadata committed, file never written); a retry inserts a second row instead of fixing the first, and _read_outreach_message_text returns None identically for this case and for a file genuinely deleted later, so the UI can't tell them apart.
- (decision) Investigated: outreach_messages lives in jobs/outreach_db.py's own SCHEMA string (no auto-migration for pre-existing DBs, unlike jobs.db's jobs table which has _ensure_columns()) - needs its own migration path, not a naive reuse of jobs.db._ensure_columns since that function's paren-parsing assumes one CREATE TABLE per schema_sql string and outreach_db.py's SCHEMA has two (contacts + outreach_messages).
- (decision) Two write sites hit this identically: jobs/cli.py's _draft_and_store_outreach (CLI) and jobs/ui_actions.py's draft_and_save_outreach (UI) - both catch (OSError, ValueError) from _atomic_write_text after insert_outreach_message already committed. Two display sites: views/jobs_list.py and views/intake.py's Message-history expanders, both already have the full outreach_messages row in scope at display time.
- (capability) CAP-1: The moment a CAP-3-style write failure happens (message text logged to the DB but the file write then fails), the outreach_messages row is marked as a known write-failure, distinct from a row whose file is simply missing for some other reason. Success: after a simulated write failure, the row's write-failure marker is set; a row that never failed has it unset.
- (capability) CAP-2: The Message-history expander (views/jobs_list.py, views/intake.py) shows a distinguishing caption for a known write-failure row instead of the generic (message file not found) caption used for every other missing-file case. Success: a row with the write-failure marker set renders the distinguishing caption; a row with a missing file but no marker still renders the existing generic caption unchanged.
- (constraint) The write-failure marker must be set on a pre-existing DB too, not just a freshly created one - outreach_db.py's ensure_schema() has no auto-migration for pre-existing DBs (unlike jobs.db), so this needs its own targeted migration (a PRAGMA table_info check + ALTER TABLE ADD COLUMN, matching this file's existing drop_legacy_message_column precedent), not a naive reuse of jobs.db._ensure_columns (see decision log: its parsing assumes one CREATE TABLE per schema string, outreach_db.py's SCHEMA has two).
- (constraint) Must be applied identically at both write-failure sites (jobs/cli.py's _draft_and_store_outreach and jobs/ui_actions.py's draft_and_save_outreach) - a fix at only one leaves the other's failures just as unmarked as before.
- (constraint) Must be applied identically at both display sites (views/jobs_list.py and views/intake.py) - same reasoning.
- (non-goal) Not a compensating DB-row delete or automatic cleanup of orphaned rows - matches this codebase's existing insert-only convention for outreach_messages (already a deliberate assumption logged in spec-atomic-file-writes.md).
- (non-goal) Not a retry/resume mechanism that re-attempts the failed file write for the same row - the drafted text is already gone once SystemExit fires (never persisted anywhere but the error message shown to the operator at the time), so there's nothing left to retry against; this only makes the historical row's state honest, not recoverable.
- (event) Self-validate: Pass 1 (coherence) OK - 2 capabilities each with intent+success, constraints all bend design (pre-existing-DB migration path, both write sites, both display sites), 2 non-goals, concrete success signal, no open questions (scope was fully resolved by investigation). Pass 2 (preservation): deferred-work.md claims land in SPEC.md (orphaned-row-indistinguishable-from-deleted-file complaint -> Why; marking at failure time -> CAP-1; UI distinguishing caption -> CAP-2; insert-only convention + no-retry-mechanism -> Non-goals). No wrapper-only content dropped. SPEC.md finalized.
