- (note) Source: deferred-work.md entry under spec-force-regenerate-control.md - neither generate_tailored_application (jobs/tailor.py:73) nor generate_paragraph_edits (docx_tailor.py:132) wraps its genai.Client() call in a try/except; a network/API error or a malformed-JSON pydantic ValidationError from model_validate_json propagates raw. views/intake.py and views/jobs_list.py only catch except SystemExit, so this surfaces as an unhandled Streamlit traceback.
- (note) Reachability confirmed by reading code: generate_tailored_application is called from jobs/cli.py's _get_or_generate_tailor_text (line 336, docx cache-miss path) and _cmd_tailor (line 377, plain-text path); generate_paragraph_edits is called from _tailor_docx_for_job (line 490). All three feed into ui_actions.py's generate_tailored_docx_for_job, the function both Streamlit views call. The force-regenerate-control spec widened exposure: Regenerate used to be a guaranteed cache hit (never called the LLM again); now it can.
- (note) Established convention confirmed: this codebase's user-facing failure pattern is raise SystemExit(<message>) (e.g. _require_raw_resume_text at jobs/cli.py:353), not a new exception class per failure type. project-context.md documents this exact gap as a 'Known gap (not yet fixed, just documented)' for extract.py/tailor.py specifically. OutreachLengthError is the only structured-exception precedent in the codebase and it carries extra data (draft_text/char_count/limit) for a UI feature this failure mode doesn't need.
- (note) No existing call site in this codebase catches google-genai SDK exceptions (checked outreach.py's draft_outreach_message too - it has zero try/except around its own client.interactions.create() call). No established precedent for which exception type(s) the SDK raises on network/API failure.
- (capability) CAP-1 intent: A Gemini API/network failure or malformed-JSON response during resume+cover-letter text generation (generate_tailored_application) is caught and re-raised as a friendly SystemExit-style message instead of propagating as a raw, unhandled exception. success: mocking the interactions.create call to raise (simulating a network/API error) or to return JSON that fails TailoredApplication.model_validate_json both result in a SystemExit reaching the caller (_cmd_tailor, _get_or_generate_tailor_text) with a message identifying what failed; existing tests mocking a successful call remain unaffected.
- (capability) CAP-2 intent: The same failure handling applies to generate_paragraph_edits (the docx paragraph-rewrite path). success: mocking its interactions.create call to raise, or to return JSON that fails TailoredResumeEdits.model_validate_json, both result in a SystemExit reaching the caller (_tailor_docx_for_job) with a message identifying what failed; existing tests mocking a successful call remain unaffected.
- (constraint) Must reuse this codebase's existing raise SystemExit(str) convention for user-facing failures, not introduce a new exception class - matches _require_raw_resume_text and every other CLI-layer failure site; the two views already catch except SystemExit at every call site that reaches these functions (ui_actions.generate_tailored_docx_for_job's own docstring confirms this), so no view-layer change is needed or in scope.
- (constraint) Must preserve the client: Optional[genai.Client] = None keyword parameter and its current signature/return type on both functions - project-context.md's Mock Usage rule requires this shape stay intact for Gemini-call mockability in tests.
- (constraint) Success-path behavior and return values are unchanged - only the failure path gains handling; a successful generation must behave identically to today, byte for byte.
- (non-goal) No retry/backoff or resilience engineering for transient network failures - conversion to a friendly error only, not making the call more likely to succeed.
- (non-goal) jobs/extract.py is out of scope - project-context.md's Known-gap note names it alongside tailor.py, but this deferred-work entry named only generate_tailored_application and generate_paragraph_edits; extract.py is a separate future item.
- (non-goal) No new structured exception class mirroring OutreachLengthError - this failure mode has no extra data worth carrying beyond a message, unlike OutreachLengthError's draft_text/char_count/limit.
- (non-goal) No change to argparse's own top-level exception handling or process exit-code behavior for the CLI entry point - already works via SystemExit's normal propagation.
- (assumption) Assumed a plain raise SystemExit(f'...: {exc}') (or equivalent message naming the failure) is sufficient user-facing text - no design request for retry buttons or structured recovery data.
- (question) What specific exception type(s) does google-genai's client.interactions.create() raise for network/API failures (rate limit, auth, timeout, 5xx)? No call site in this codebase catches or documents this today - worth checking the installed google-genai version's source/docs before implementing, to catch precisely rather than a bare except Exception.
