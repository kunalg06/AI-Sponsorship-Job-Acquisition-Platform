---
topic: error_display_text should surface a chained exception's message when the top-level exception's str() is empty
updated: 2026-07-17T09:43
---

- (decision) Source: deferred-work.md entry under spec-error-display-hardening.md - error_display_text's fallback (f"{type(exc).__name__}: (no error message)") reads like an internal repr and ignores __cause__/__context__ on a chained exception.
- (capability) CAP-1: When str(exc).strip() is empty, check exc.__cause__ then exc.__context__ (in that order) for a non-empty str() and surface that instead of the bare type-name fallback. Success: an exception with empty str() but a __cause__ whose str() is non-empty renders that cause's message via error_display_text, not the type-name fallback.
- (constraint) Walk the chain (cause, then its own cause/context, ...) rather than stopping at one level, since a chain can be more than one exception deep; stop at the first link with a non-empty str().
- (constraint) If no link in the chain (including the original exc) has a non-empty str(), keep the existing type-name fallback, now naming the innermost exception in the chain (the most specific one) rather than always the outermost.
- (constraint) Must not raise or infinite-loop on a cyclical __context__ chain (Python allows exc.__context__ cycles via 'raise ... from' patterns/manual construction) - cap traversal or track seen ids.
- (constraint) Reuse error_display_text's existing try/except around str(exc) for each link walked, since any exception type's __str__ can itself raise (already handled for the top-level exc).
- (non-goal) Not a redesign of error presentation or copy tone beyond this - no new UI, no prefixing chained messages with explanatory text like 'caused by:' unless that turns out to be needed for clarity (open question).
- (non-goal) Not touching the 10 call sites in views/*.py - they already call error_display_text(exc); this only changes that function's internal fallback logic.
- (question) Should a surfaced chained message be prefixed (e.g. 'ExcType: (no message) - caused by: <cause message>') to avoid looking like the outer exception's own text, or returned bare? Leaning bare for minimalism, matching this function's existing plain-passthrough behavior for a normal str(exc) - decide during implementation, low-stakes copy choice.
