---
topic: extract_text_from_docx should read paragraph/table content wrapped in w:sdt content controls
updated: 2026-07-17T09:56
---

- (decision) Source: deferred-work.md entry under spec-resume-docx-table-extraction.md - extract_text_from_docx silently drops w:sdt-wrapped paragraphs/tables; confirmed via python-docx source that CT_Body.inner_content_elements uses xpath ./w:p | ./w:tbl (direct children only), so w:sdt is skipped even by the public iter_inner_content() API this codebase already uses.
- (capability) CAP-1: extract_text_from_docx reads paragraph and table content wrapped in one or more levels of w:sdt (content control), in document order alongside ordinary paragraphs/tables, instead of silently dropping it. Success: a .docx with a resume section wrapped in a single content control (as Word's own resume templates commonly do) has that section's text appear in extract_text_from_docx's output.
- (capability) CAP-2: nested w:sdt (a content control inside another content control) is also read, up to a bounded depth, walked iteratively rather than via Python recursion. Success: a .docx with a paragraph nested two levels deep inside w:sdt wrappers has that paragraph's text appear in the output; a pathologically/adversarially deep w:sdt chain (beyond the bound) does not crash or hang extraction (no RecursionError, no unbounded work) - content beyond the bound is simply not read.
- (constraint) Walk must be iterative (explicit stack/queue), not Python recursion, so a deeply/adversarially nested w:sdt chain can't raise RecursionError or degrade performance unboundedly.
- (constraint) Depth must be capped at a fixed bound; content nested beyond the bound is silently skipped (matches this codebase's existing 'silent content loss for rare structures' precedent - headers/footers/text boxes are already out of scope for the same reason).
- (constraint) Must preserve existing document-order interleaving behavior (the CAP-1/CAP-2 fix this builds on, from spec-resume-docx-table-extraction.md) - an sdt-wrapped paragraph/table appears in the output at its actual position in the document, not appended at the end.
- (constraint) Must reuse the existing _row_text table-row handling (merged-cell dedup, multi-paragraph newline flattening, all-blank-row skip) for any table found inside a w:sdt, not a separate/divergent code path.
- (non-goal) Not headers, footers, or text boxes - already out of scope per spec-resume-docx-table-extraction.md, unchanged by this spec.
- (non-goal) Not w:sdt elements nested inside other wrapper types (w:ins, w:customXml, etc.) - only w:sdt itself is in scope; other wrapper elements remain a separate, still-open gap (inner_content_elements' own docstring notes w:ins is excluded too).
- (assumption) Assumed the depth bound can be a fixed constant (e.g. 20) chosen for adversarial-input safety, not measured against real Word documents - real CV content controls are essentially never nested more than 1-2 levels deep, so the bound is a defensive ceiling, not a tuned value.
- (event) Self-validate: Pass 1 (coherence) OK - 2 capabilities each with intent+success, constraints all bend design (iterative-not-recursive, depth cap, document-order preservation, _row_text reuse), 2 non-goals, concrete success signal, 1 assumption (unmeasured defensive depth constant). Pass 2 (preservation): all deferred-work.md claims land in SPEC.md (w:sdt drop complaint + python-docx xpath confirmation -> Why; read wrapped content -> CAP-1; nested/bounded/iterative -> CAP-2; headers/footers/other-wrappers exclusion -> Non-goals). No wrapper-only content dropped. SPEC.md finalized.
