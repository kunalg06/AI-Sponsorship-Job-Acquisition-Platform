topic: Extend extract_text_from_docx to also read table content, in document order
updated: 2026-07-16T23:17
---

- (note) Source: deferred-work.md entry under spec-admin-cv-upload-resume-registration.md - extract_text_from_docx (src/resume/extract.py:56-58) only reads doc.paragraphs; tables, headers/footers, and text boxes are silently dropped, so a table-based CV layout (skills/dates commonly laid out in tables) produces an incomplete extracted-text string with no error or warning. Feeds into the Admin page's CV-upload -> resume.extract.extract_profile -> resume.db.insert_profile pipeline.
- (decision) Narrowed scope to tables only, not headers/footers/text boxes (also named in the original deferred item) - tables are the common CV case (skills/dates grids); headers/footers need a different API (document.sections[i].header/footer) and text boxes are embedded XML shapes not readily read via python-docx at all. This was a deliberate frozen-spec scope choice originally ('pure python-docx paragraph-join'); extending it to also read tables is purely additive and behavior-tightening (extracts more real text, never less), so it does not need destructive renegotiation.
- (note) Verified empirically against a real generated .docx: walking document.element.body's children in order and dispatching on child.tag (docx.oxml.ns.qn('w:p') vs qn('w:tbl')), wrapping w:tbl children as docx.table.Table(child, document) to read row.cells, correctly preserves original document order - paragraphs and tables interleave exactly as authored, not paragraphs-then-tables.
- (capability) CAP-1 intent: extract_text_from_docx includes table content (every row's cell text) in its returned text, in the same relative position tables appear in the source document, so a table-based CV layout no longer silently loses skills/dates/etc. laid out in a table. success: a .docx with a paragraph, then a table, then another paragraph produces extracted text containing all three pieces of content in that order, with each table row's cells present (joined by a separator) and no cell content missing.
- (constraint) The existing test (test_extract_text_from_docx_joins_paragraph_text_with_newlines, a table-free .docx) must keep passing unchanged - its exact current output for a paragraph-only document is unaffected by this change.
- (constraint) Table rows join their cells with a separator (' | ') distinct from the newline used between paragraphs/rows, so the LLM consuming this text can still tell where one row ends and the next begins.
- (constraint) extract_text_from_docx's signature and return type (str) are unchanged - no new parameters, no new exceptions introduced beyond what python-docx itself can already raise for a malformed .docx.
- (non-goal) Headers, footers, and text boxes remain out of scope - split off as a separate, still-open deferred item (different APIs, rarer in CVs than tables).
- (non-goal) Nested tables (a table inside a table cell) are not specially handled - python-docx's cell.text does not recurse into a nested w:tbl, so nested-table content stays dropped. Rare enough in real CVs not to warrant added complexity here.
- (non-goal) No change to resume.extract.extract_profile, resume.db.insert_profile, or the Admin page UI - this is a pure input-completeness fix to the text-extraction step only.
- (non-goal) No warning/error surfaced when a document has zero tables or zero paragraphs - matches the function's existing silent, best-effort extraction behavior.
