topic: Server-side stderr diagnostic for Gemini call failures before conversion to SystemExit
updated: 2026-07-16T23:06
---

- (note) Source: deferred-work.md entry under spec-tailoring-llm-error-handling.md - the except blocks in generate_tailored_application (jobs/tailor.py) and generate_paragraph_edits (docx_tailor.py) convert every Gemini failure straight to SystemExit with no server-side record of the original exception/traceback; once views/*.py catch it and show st.error(error_display_text(exc)), the full traceback is gone, leaving near-zero diagnosability for a real production failure.
- (note) Confirmed via grep: this codebase has zero usage of Python's logging module anywhere (src/, app.py, views/). The only existing precedent for a durable diagnostic is app.py:31-36's plain print(<message>, file=sys.stderr) for the missing-GEMINI_API_KEY case. Neither jobs/tailor.py nor jobs/docx_tailor.py currently imports sys or traceback.
- (decision) Use traceback.print_exc(file=sys.stderr) inside the except block, before raise SystemExit(...) from exc, rather than introducing the logging module - matches this project's only existing diagnostic-output convention (plain stderr print) instead of adding a new one. Prints the full original traceback, not just a message, directly answering the deferred item's stated problem (the traceback is gone once shown to the user).
- (capability) CAP-1 intent: When generate_tailored_application (jobs/tailor.py) catches a Gemini failure, the original exception's full traceback is written to stderr before conversion to SystemExit, so a server operator can diagnose a real production failure even though the Streamlit user only sees the friendly SystemExit message. success: mocking the Gemini call to raise any of the already-caught types (APIError, UnknownApiResponseError, httpx.HTTPError, ValidationError, RuntimeError) results in the original exception's traceback text (e.g. the exception's type name and message) appearing on stderr, in addition to the existing SystemExit behavior being unchanged.
- (capability) CAP-2 intent: The same stderr-traceback behavior applies to generate_paragraph_edits (docx_tailor.py). success: same as CAP-1, verified against generate_paragraph_edits's own except block and its own caught exception types.
- (constraint) Use traceback.print_exc(file=sys.stderr) (stdlib traceback module) - do not introduce the logging module, matching this codebase's sole existing diagnostic-output precedent (app.py's plain print(..., file=sys.stderr)).
- (constraint) SystemExit's message, the except tuple's exception types, and every other existing behavior of both functions are unchanged - this is additive-only (a new side effect on the failure path), not a refactor.
- (constraint) The stderr write must happen inside the except block (while the original traceback context is still live via sys.exc_info()), before the raise SystemExit(...) from exc line.
- (non-goal) No introduction of the logging module, log levels, log rotation, or any structured-logging framework - plain stderr is the deliberate, minimal fix matching this project's existing convention and its no-CI/no-observability-infra-by-design posture.
- (non-goal) No change to jobs/extract.py or jobs/outreach.py - out of scope, same as the parent spec (spec-tailoring-llm-error-handling.md) that deferred this item.
- (non-goal) No change to what the Streamlit user sees (st.error(error_display_text(exc)) output is unaffected) - this is purely a server-side diagnostic addition.
- (non-goal) No log aggregation, remote log shipping, or persistence beyond the process's own stderr stream - out of scope for a single-user local tool with no deployment infrastructure for that.
