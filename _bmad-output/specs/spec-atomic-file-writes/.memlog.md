- (direction) Source: deferred-work.md, 2 entries flagged in 2026-07-12 round-2 adversarial+edge-case review of spec-tailored-content-file-only-storage.md and the 2026-07-12 review of spec-outreach-file-only-storage.md, both citing the same root cause: no write path in jobs/cli.py uses a temp-file-then-rename (atomic replace) pattern.
- (capability) CAP-1: tailored resume/cover-letter output writes become atomic (temp-file-then-rename), so a crash or failure mid-write never leaves a partial file that still passes this codebase's 'does it exist' cache/already-generated checks. Source: jobs/cli.py:242-243 (primary tailoring path) and jobs/cli.py:535,539 (legacy-migration/regeneration path) -- same pair of writes, two call sites.
- (capability) CAP-2: the legacy-tailoring DB-text-backup write becomes atomic for the same reason -- a partial backup file is worse than no backup file, since it would look present but be truncated/corrupt. Source: jobs/cli.py:663.
- (capability) CAP-3: the outreach-message draft write becomes atomic, and a failure there is caught and surfaced with a message that explicitly distinguishes 'the drafted text was lost even though its metadata was saved' from other error types -- this write runs *after* insert_outreach_message's DB commit already succeeded, so an uncaught/generic failure here leaves a DB row (channel, contact_name, char_count) permanently pointing at text that no longer exists anywhere, with no specific indication of that mismatch. Source: jobs/cli.py:794, insert_outreach_message commit at jobs/cli.py:784-786.
- (constraint) A single, shared, reusable atomic-write mechanism must be used across all four write sites (CAP-1's two call sites, CAP-2, CAP-3) -- not four independent implementations. Rules out ad hoc per-site fixes.
- (constraint) The temp file must be written in the same directory/filesystem as the final target so os.replace()/rename is atomic -- a cross-filesystem rename silently degrades to copy+delete, defeating the point. A precedent for this exact pattern already exists in this project: _bmad/scripts/memlog.py's own write_atomic() helper (temp + flush + fsync + os.replace, same directory).
- (constraint) No new dependencies -- Python stdlib (tempfile or a simple Path.with_suffix('.tmp') sibling, plus os.replace) is sufficient.
- (constraint) Existing encoding (utf-8) and existing directory-creation behavior (path.parent.mkdir(parents=True, exist_ok=True)) at each site must be preserved -- the fix targets write atomicity only, not encoding or directory setup.
- (assumption) No compensating delete of the outreach_messages row on a post-commit write failure (CAP-3) -- matches this codebase's established insert-only convention for its other tables (jobs, profiles). The fix prioritizes a clear, specific error message over transactional rollback, since adding a delete path is a bigger behavioral change than this spec's scope.
- (non-goal) Not adding atomic-write handling to any file-write path outside jobs/cli.py's four identified sites (e.g. not resume/db.py, register/db.py, or any other module) -- scope is limited to the sites the deferred-work entries named.
- (non-goal) Not adding a compensating DB-row delete/rollback for the outreach-write-failure case -- see the logged assumption above.
- (non-goal) Not adding retry/backoff logic for transient write failures -- a single clean failure-and-report is sufficient; not building resilience beyond atomicity.
- (non-goal) Not changing what data is written at any site -- only how it's written (atomically vs. directly).
