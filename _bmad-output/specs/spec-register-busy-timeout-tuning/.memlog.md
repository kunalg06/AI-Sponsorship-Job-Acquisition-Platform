---
topic: Measure and tune register/db.py's busy_timeout against the real replace_all() operation it protects
updated: 2026-07-16T23:33
---

- (note) Source: deferred-work.md entry under spec-admin-destructive-safety.md - register/db.py's connect() sets PRAGMA busy_timeout=5000 (db.py:70), copied from mcp_server/tools.py's convention (used for small single-row operations) without ever measuring it against replace_all()'s actual DELETE+INSERT of the ~142k-row sponsor register inside one transaction (db.py:75-103).
- (note) Measured directly against the real local data/sponsors.db (142,235 rows - matches the '~142k' the deferred item names): 3 trials of replace_all() against a temp-copy DB took 553.7ms, 614.9ms, 635.3ms. Current 5000ms value already has a healthy ~8x safety margin locally.
- (note) busy_timeout governs how long a concurrent reader/writer waits for replace_all()'s write lock to release, not how long replace_all() itself is allowed to run - so the real question is whether a contending connection survives a slow replace_all() on unknown infrastructure. This project has an aspirational Streamlit Cloud deploy target (data/*.db is gitignored, so a fresh cloud deploy's Admin 'Refresh sponsor register now' button runs this exact 142k-row replace_all() for real on unknown-performance shared infra) - an unmeasured, environment-unaware guess is genuinely fragile even though the measured local baseline is comfortable.
- (decision) Bump only register/db.py's busy_timeout, not mcp_server/tools.py's or resume/db.py's - those two protect much smaller single-row operations (mark applied/discarded, insert profile) that don't share replace_all()'s risk profile. Diverging register/db.py from the previously-shared 5000ms convention is now evidence-based rather than an inconsistency.
- (decision) Target value: 15000ms (~24x the measured 550-635ms baseline). Reasoning: busy_timeout only costs anything during actual lock contention (rare - single-user tool, and the only writer is the ingest button); a generous margin has no downside when there's no contention, and 24x comfortably covers slower/virtualized cloud disk I/O or future register growth without needing to guess a 'perfect' number.
- (capability) CAP-1 intent: register/db.py's busy_timeout value is backed by a real, documented measurement of replace_all() against a realistically large register (order of 100k+ rows), not an unmeasured copy of an unrelated call site's convention. success: a test seeds a synthetic dataset of at least 100k SponsorRecord rows, measures replace_all()'s wall-clock duration, and asserts it completes in well under the configured busy_timeout (order-of-magnitude margin, not a hair's-breadth pass) - catching a future ingest-performance regression before it erodes the safety margin.
- (capability) CAP-2 intent: register/db.py's busy_timeout is set to a value with an explicit, comfortable, documented safety margin over the measured baseline, specifically for the large-scale replace_all() operation it protects. success: connect()'s PRAGMA busy_timeout value is verifiably higher than the previous 5000ms (e.g. 15000ms), with an inline comment stating the measured baseline and the reasoning for the chosen margin; existing test_connect_sets_busy_timeout_pragma is updated to assert the new value.
- (constraint) Only register/db.py's connect() changes - mcp_server/tools.py's and resume/db.py's busy_timeout=5000 stay untouched, since they protect different, much smaller operations.
- (constraint) The new performance test must not depend on network access or the real gov.uk CSV source - generate synthetic SponsorRecord rows locally (matches this codebase's existing test convention of never hitting real external services in tests).
- (constraint) The chosen busy_timeout value and its reasoning must be visible in code (inline comment), not just in this spec - a future reader of db.py shouldn't need to find this spec to understand why the value is what it is.
- (non-goal) No change to the DELETE+INSERT transaction shape itself (e.g. batching, WAL mode, incremental diff-based updates) - this spec is only about the busy_timeout value being evidence-based, not about making replace_all() itself faster or structurally different.
- (non-goal) No real end-to-end measurement against actual Streamlit Cloud infrastructure - that's not available to test from here; the safety margin is a principled buffer over the real local measurement, not a guarantee for every possible environment.
- (non-goal) No change to the busy_timeout PRAGMA usage in mcp_server/tools.py or resume/db.py.
