---
topic: make DEFAULT_GENERATED_CV_DIR configurable via a GENERATED_CV_DIR env var so CLI --out-dir defaults and the UI agree
updated: 2026-07-17T11:05
---

- (decision) Source: deferred-work.md entry under spec-outreach-file-only-storage.md - CLI outreach/follow-up/tailor-docx commands accept --out-dir, but the UI always reads/writes at the hardcoded DEFAULT_GENERATED_CV_DIR, so a message drafted via a custom --out-dir shows as file-not-found in the UI.
- (decision) Investigated: DEFAULT_GENERATED_CV_DIR (jobs/cli.py:77) is a single module-level constant that every --out-dir default and every UI read/write site already imports and uses unmodified (confirmed via grep, no site hardcodes a duplicate string). load_dotenv() already runs at cli.py module level (line 70) before this constant is defined, matching the exact pattern already used for GEMINI_API_KEY in app.py.
- (decision) Design settled by investigation, not escalated further: read DEFAULT_GENERATED_CV_DIR from a GENERATED_CV_DIR env var (falling back to the current "cv/generated_cv"), so setting it once in .env makes the CLI's own --out-dir defaults and every UI site agree - closes the gap for the common case with no settings UI and no per-invocation override plumbing.
- (capability) CAP-1: DEFAULT_GENERATED_CV_DIR reads from a GENERATED_CV_DIR environment variable when set, falling back to the current "cv/generated_cv" default when unset. Success: with GENERATED_CV_DIR set in the environment, jobs.cli.DEFAULT_GENERATED_CV_DIR equals that value; with it unset, the constant is unchanged ("cv/generated_cv").
- (capability) CAP-2: A single GENERATED_CV_DIR env var setting makes the CLI's own --out-dir defaults (tailor-docx, outreach, follow-up, migrate-legacy-outreach) and every UI read/write site consistent, without any of them needing individual changes. Success: with GENERATED_CV_DIR set, a job tailored via the CLI's default --out-dir produces files the UI finds at the same path, and vice versa.
- (constraint) Must not change the current default behavior when GENERATED_CV_DIR is unset - existing installs with no .env entry for it must resolve to the exact same "cv/generated_cv" path as today.
- (constraint) Must read the env var at the same module-load point DEFAULT_GENERATED_CV_DIR is currently defined (after load_dotenv() already runs in jobs/cli.py), not lazily per-call, so every consumer that already imports the constant as a plain string keeps working unchanged - no consumer needs to switch from importing a constant to calling a function.
- (non-goal) Not a UI settings panel or any per-session/per-user override - a single process-wide env var, matching this codebase's existing GEMINI_API_KEY convention exactly.
- (non-goal) Not making the UI follow an explicit one-off --out-dir override on a single CLI invocation - only the shared, env-var-configured default is unified; a deliberate ad-hoc override remains a CLI-only, UI-invisible choice, same as today.
- (event) Self-validate: Pass 1 (coherence) OK - 2 capabilities each with intent+success, constraints bend design (no-behavior-change-when-unset, module-load-time not lazy), 2 non-goals, concrete success signal. Pass 2 (preservation): deferred-work.md claims land in SPEC.md (CLI/UI out-dir divergence -> Why; env-var-configurable constant -> CAP-1; single-setting consistency -> CAP-2; no settings UI, no ad-hoc-override-following -> Non-goals). No wrapper-only content dropped. SPEC.md finalized.
