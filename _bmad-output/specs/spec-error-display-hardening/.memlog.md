topic: Harden st.error(str(exc)) across all Streamlit views so no exception ever renders a blank error box
updated: 2026-07-15T22:45
---

- (decision) Introduce one shared helper (e.g. jobs.ui_actions._show_error or a small views-layer helper) that wraps st.error, falling back to an informative message (exception type name + repr, not just str()) when str(exc) is empty. Route all 10 existing call sites through it, replacing st.error(str(exc)) verbatim.
- (capability) CAP-1: A caught exception with a non-empty str() displays exactly as it does today (byte-identical message) -- no behavior change for the common case. Success: existing behavior preserved, verified by a test asserting the helper returns str(exc) unchanged when non-empty.
- (capability) CAP-2: A caught exception with an empty str() (e.g. some OSError/HTTPError subclasses constructed with no message) displays an informative fallback identifying the exception type, never a blank st.error box. Success: a test constructing an exception with empty str() asserts the helper's output is non-empty and contains the exception's class name.
- (capability) CAP-3: All 10 existing call sites (admin.py:50,76,128,195; intake.py:369,455; jobs_list.py:105,124,288,374) are routed through the shared helper, replacing the bare st.error(str(exc)) pattern. Success: grep for st.error(str(exc)) across views/ returns zero matches; the full pytest suite still passes.
- (constraint) Helper lives in src/jobs/ui_actions.py (already imported by all 3 view files per project-context.md's 'views import CLI-layer helpers directly' convention) rather than a new views/ module -- matches existing layering, no new import surface.
- (constraint) Fallback message must not swallow or reformat a non-empty str(exc) -- only the empty-string case changes rendering. This is purely a defensive fallback, not a redesign of error presentation.
- (assumption) 6 of the 10 sites catch SystemExit raised by this codebase's own CLI-layer helpers (already-rich embedded messages, e.g. the atomic-write failure text) -- lower real-world risk than the 4 Exception-catching sites in admin.py (network/DB/Gemini calls that can raise bare OSError/HTTPError). Hardening all 10 uniformly anyway, since it's one mechanical fix and consistency matters more than scoping to the higher-risk subset.
- (direction) Non-goal: not changing exception handling/catching logic (which exception types are caught, whether more specific except clauses are added) -- only what gets displayed once an exception is already caught. Not adding logging, not changing which sites catch Exception vs SystemExit vs a narrower type.
